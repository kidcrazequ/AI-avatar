## 适用问题特征

用户问题命中以下任一形态时，**必须先加载本技能再回答**：

| 形态 | 通用样例 |
|---|---|
| 为什么 X 没做 Y | 「为什么 <项目/产品> 没采用 <方案>？」 |
| 为什么选了 Z 而不是 W | 「为什么 <方案 A> 而不是 <方案 B>？」 |
| 这个决策当时怎么定的 | 「<某项配置/参数> 当时怎么定的？」 |
| 当时为什么取消了 X | 「当时为什么取消了 <某功能/某要求>？」 |
| 谁拍板的 / 谁讨论过 | 「<某决策> 谁拍板的？」 |

**反例**（不属本技能，走数据查询 / 常规检索）：
- 「<X> 是什么 / 容量多少 / 参数是」（事实题，不是回溯）
- 「最新的 <Y> 数据」（数据查询，不是回溯）

## 核心流程（5 步，禁止跳步）

### 步骤 1：候选文件清单（search_knowledge × 2）

并行调两次 `search_knowledge`，覆盖**两个独立检索维度**：

```js
// 维度 A：以项目/方案/阶段为关键词
search_knowledge({ query: "<项目阶段> 变更点 评审" })
// 维度 B：以被询问对象 + "评估 / 是否使用 / 讨论 / 决策" 为关键词
search_knowledge({ query: "<被询问对象> 评估 是否使用 讨论" })
```

> **失败处置**：任一次 `search_knowledge` 返回过长（触发 spool 落盘提示），**必须**用 `read_tool_result` 读 spool 完整文件，不要凭头/尾摘要作答。`read_lines` 会因路径不在工作区而失败，必须用 `read_tool_result`。

预期产物：一份候选文件清单（典型 3-5 份），按相关性排序。

### 步骤 2：识别表格类决策记录（query_excel）

候选清单中如果有「**输入确认清单**」「**评审记录**」「**变更点表**」「**讨论纪要**」**任一**，**必须**用 `query_excel` 精确取行——这类表格的关键证据（拍板人答复、技术要求、风险注释）通常在备注列，被 search_knowledge 的摘要算法切掉。

```js
// 先看 schema 知道有哪些列、哪些 sheet
query_excel({ file: "<文件名>", schema: true })
// 再按"涉及对象"过滤取行（不要 columns 限制，让备注列出来）
query_excel({
  file: "<文件名>",
  sheet: "<sheet>",
  filter: { "<可能列名>": { "$contains": "<被询问对象>" } }
})
```

### 步骤 3：旁证（search_knowledge × N）

把步骤 2 拿到的决策点（如"防凝露""配置不统一"等具体考量）作为新关键词，回查 search_knowledge 找**佐证文件**（用户手册、对外资料、其他项目同类讨论）。这一步证明"决策不是孤立的"。

### 步骤 4：综合证据，结构化回答

回答按下述**强制结构**输出（违反结构判定为偷懒）：

```markdown
**<阶段> 对 <被询问对象> 做过专项评估，最终 <结论>，核心考量如下：**

**1. <考量点 1 名称>**
<具体证据：人名 / 料号 / 原文片段 / 数值>。<决策含义>。

**2. <考量点 2 名称>**
...

**3. <考量点 3 名称>**
...

（一般 3-5 点；少于 3 点说明检索不够深）

综合以上因素，<阶段> <最终采用方案>，避免 <被规避的风险>。
```

### 步骤 5：来源声明

必须在末尾列出**具体文件名**（不是 `_excel/*.json` 中间产物路径）：

```
来源：
· `<原始 Excel 文件名>.xlsx`（<拍板人/关键决策人>逐行答复）
· `<项目变更点或评审材料>.docx / .pptx`（同期决策背景）
· `<用户手册或对外资料>.docx`（旁证）
```

**禁止用 `_excel/*.json` 路径作为来源**——那是内部中间产物，用户看不懂；自动 normalize 已映射为原 .xlsx 路径，引用时直接用原文件名。

## 反模式（禁止）

下列回答模式会被判定为**人格失败**：

❌ **泛词替代原文**：「<X 方案>侧重高<指标>，<Y 方案>针对<场景>」——没有具体料号 / 决策人 / 项目阶段；
❌ **只列 1-2 个考量**：决策回溯类问题至少 3 个考量，少于 3 点说明步骤 1-3 没走通；
❌ **只用 search_knowledge 不用 query_excel**：备注列证据丢失，回答会浮于表面；
❌ **来源指 `_excel/*.json`**：暴露内部缓存路径，应映射回原 xlsx 文件名；
❌ **跳过 read_tool_result 直接读 spool**：用 `read_lines` 会路径穿越失败。

## 与其他技能的关系

- 调用前置：无（用户问题命中触发即可加载）
- 调用后置：如果回答需要画图或对比，可串 `load_skill('chart-from-knowledge')`
- 互斥：与领域数据查询技能（如 `query-product-quality`）互斥——那类是数据查询，本技能是决策回溯
