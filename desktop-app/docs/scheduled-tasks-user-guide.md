# Scheduled Tasks 用户指南

> 让分身按 cron 表达式自动给你发消息 —— 每天的早安、每周的工作复盘、每小时的汇率提醒，都可以由分身自动跑起来。

**入口**：桌面端导航栏 → **「定时」** 面板。

---

## 一、快速上手（30 秒）

1. 选中一个分身，点击导航栏**「定时」**进入面板
2. 点右上角 **「+ 新建」**
3. 填写：
   - **任务名称**：如「每日早安」
   - **触发频率**：简易模式下选「每天」并填 09:00
   - **提示词内容**：触发时发送给分身的消息，如：「写一句今天的早安祝福」
4. 点 **「创建」**

完成后，每天 09:00 分身会自动收到这条消息，并按你的人设、知识库、工具来回复。

---

## 二、触发频率：简易 vs 高级

### 简易模式

适合 80% 场景：

| 模式 | 含义 | 示例 |
|------|------|------|
| 每天 | 每天某时某分 | 09:00 → `0 9 * * *` |
| 每周 | 每周某天某时某分 | 周一 09:00 → `0 9 * * 1` |
| 每月 | 每月某日某时某分 | 每月 1 日 09:00 → `0 9 1 * *` |
| 每 N 小时 | 每天 0 点起算每 N 小时 | 每 6 小时 → `0 */6 * * *` |

### 高级模式（cron 表达式）

完整的 5 字段 cron：`分 时 日 月 周`。常用例：

| 表达式 | 含义 |
|--------|------|
| `*/15 * * * *` | 每 15 分钟 |
| `0 */2 * * *` | 每两小时整点 |
| `30 9 * * 1-5` | 工作日（周一到周五）09:30 |
| `0 0 1 1 *` | 每年元旦 0:00 |

**精度说明**：cron 精度到分钟，不支持「每 30 秒」之类的秒级触发；这是 cron 标准本身的限制。

**实时预览**：表单内会用中文实时翻译你的表达式（例如「`30 9 * * 1-5`」→「在 09:30 - 仅在周一至周五」），并显示**下 3 次触发的本地时间**，方便你确认是否符合预期。

## 三、可选配置

### 时区（IANA timezone）

默认 `Asia/Shanghai`。出差或团队跨时区可改为 `Europe/London`、`America/Los_Angeles` 等。注意 cron 表达式中的「时」是指本时区时间。

### 对话 ID（高级）

- **留空（推荐）**：每次触发**新建一条对话**，标题自动命名 `[Schedule] 任务名 @ YYYY-MM-DD HH:MM`，便于在侧栏识别。
- **填入已有对话 ID**：所有触发的消息追加到这条对话里，适合「持续日报」场景。注意：如果该对话被删除，下一次触发会自动降级为「新建对话」。

### 启用 / 禁用

不想暂时触发但又想保留任务，用 **「⏸ 停用」**；想恢复点 **「▶ 启用」**。停用期间错过的触发**不会补跑**（避免重启时一次性涌入大量历史消息）。

---

## 四、历史日志

每次触发都会留一条记录，可见：

- **时间戳**：本地时区
- **状态**：成功 / 失败 / 错过 / 运行中
- **对话 ID**：实际写入或新建的对话
- **耗时**：从触发到分身回复完成的端到端毫秒
- **错误信息**：失败时的简短诊断（最多 500 字）

最近 100 条触发会显示在面板右侧。删除任务时历史日志会一并删除。

---

## 五、立即触发（调试）

每个任务右上角有 **「▶ 立即触发」** 按钮，跳过 cron 等待直接跑一次，适合：

- 调试 prompt 是否能让分身按预期回复
- 验证某条任务真的能跑通
- 不影响下一次到点触发（不重置计时器）

注意：同一秒内重复触发会被幂等约束拦截（避免重复发同样消息）。

---

## 六、安全与权限

定时任务触发的消息会以 `trustTier='proxy'` 进入分身：

- 与 **Plan / Permission Mode**（#7）共享同一套灰名单拦截规则
- 高危工具（如 `delete_file`、`run_command`）默认会被拦截，避免无人值守时执行破坏性操作
- 普通对话工具（如知识检索、笔记追加）正常放行

如需放宽，请到对应分身的「权限」配置中显式允许相应工具。

## 七、常见问题

### Q1：为什么 09:00 的任务到 09:00:30 才触发？

croner 内部按整分钟检查并允许 < 1 秒抖动。绝大多数任务影响可忽略；如果你严格要求秒级精度，建议使用专用的事件总线而非 cron。

### Q2：电脑关机时错过了触发，会补跑吗？

**不会**。这是设计取舍：避免开机时一次性涌入数十条历史消息，造成 token 突发与对话紊乱。错过的会在历史日志记为 `missed`，仅作审计。

如果你需要「错过即补」语义，请人工点 **「▶ 立即触发」**。

### Q3：定时任务消息会消耗 API token 吗？

**会**，按正常对话计费。建议：

- 用 prompt 模板（话术）库管理重复内容，避免每次写长 prompt
- 频率较高的任务（如每 15 分钟）请评估月度 token 消耗

### Q4：可以让定时任务调用工具吗？

**可以**。任务消息进入分身后与正常对话同等待遇，会按分身的工具配置选择性调用工具（含 MCP）。

### Q5：删除分身后定时任务还会跑吗？

不会。定时任务以 `avatar_id` 关联分身，分身被删除后下次重启时其 schedules 会因 avatar 不存在而被跳过；建议在删除分身前手动清理对应的定时任务，保持数据整洁。

## 八、技术参考（开发者）

- 表 schema：`schedules` + `schedule_runs`（v10 引入），见 `electron/database.ts`
- 后端调度：`croner` 库，封装在 `electron/cron-scheduler.ts` 的 `scheduleCron / cancelCron / getNextRuns` 方法
- IPC 命名空间：`schedule:list / get / create / update / delete / set-enabled / trigger-now / get-next-runs / list-runs / record-run-finish`
- 渲染端处理器：`src/services/schedule-trigger-handler.ts`
- 幂等机制：`UNIQUE(schedule_id, fired_at_utc)` 唯一约束 + `recordRunStart` conflict 跳过
