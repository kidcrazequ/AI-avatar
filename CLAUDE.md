# AI 分身项目 (Soul)

这是一个 AI 分身管理项目。每个分身是一个独立的 AI 专家，拥有自己的灵魂（人格）、知识包和技能树。

## 项目结构

```
soul/
├── templates/              ← 模板（给工种专家创建新分身用）
├── shared/
│   ├── knowledge/          ← 共享知识（所有分身可引用）
│   ├── skills/             ← 公共技能（跨分身共享，唯一正本）
│   │   └── community/      ← GitHub 开源技能（由 soul-sync.sh 管理）
│   ├── design-systems/     ← 共享设计语料（品牌 design.md）
│   └── starter-components/ ← 共享 UI 组件片段
├── scripts/                ← 工具脚本（soul-sync.sh 等）
└── avatars/                ← 分身目录（每个子目录 = 一个分身）
```

## 如何使用分身

进入分身目录，启动 Claude：

```bash
cd ~/AI/soul/avatars/ci-storage-expert
claude
```

Claude 会自动加载该分身的 CLAUDE.md → soul.md → knowledge/ → memory/，以该分身的身份与你交互。

## 如何创建新分身

> **AI 辅助创建时的强制要求**：必须先读取对应模板文件再创建，禁止跳过模板凭自己理解生成内容。

### 模板关系图

```
创建一个完整分身需要以下文件，每个文件对应一个模板：

soul.md ←── templates/soul-template.md      （人格定义：身份、风格、原则）
            ↑ 撰写指南：templates/soul-guide.md

CLAUDE.md ←── templates/agent-template.md   （操作规则：知识库约束、工作流程、第一性原理）

knowledge/README.md ←── templates/knowledge-readme-template.md （知识库索引）

skills/*.md ←── templates/skill-template.md  （技能定义，公共技能在 shared/skills/）

tests/cases/*.md ←── templates/test-case-template.md （测试用例）

memory/MEMORY.md ←── 空文件                  （长期记忆）
```

### 创建步骤

1. 在 `avatars/` 下创建新目录（目录名用小写 + 连字符，如 `power-grid-expert`）
2. **读取** `templates/soul-guide.md` 了解撰写原则，然后**读取** `templates/soul-template.md` 创建 `soul.md`（建议由该工种资深工程师撰写）
3. **读取** `templates/agent-template.md` 创建 `CLAUDE.md`（包含知识库约束、第一性原理、工作流程）
4. **读取** `templates/knowledge-readme-template.md` 创建 `knowledge/README.md`，在 `knowledge/` 中放入专业知识文件
5. **读取** `templates/skill-template.md` 在 `skills/` 中定义技能。通用技能优先引用 `shared/skills/`，仅创建领域专属技能
6. 创建 `memory/MEMORY.md` 空文件
7. **读取** `templates/test-case-template.md` 创建至少 5 个测试用例（红线 2 + 知识库约束 1 + 数据溯源 1 + 人格 1）

### 核心约束（所有分身必须遵守）

- **知识库优先**：涉及专业事实、参数、数据、方案取舍、工程决策时，回答必须基于 `knowledge/` 目录；寒暄、确认、需求澄清、对话承接、格式偏好确认等非知识库问题不检索
- **数据可溯源**：关键数据标注来源文件，缺数据时诚实说明
- **第一性原理**：从本质出发分析问题，拒绝"业界通常这样做"的表面类比
- **稳定回答**：知识库事实问答 / 工程决策类回答先归并事实、判断和不确定项，再按复杂度组织；简单事实查询可压缩为"结论 + 来源"

## 使用开源技能

Soul 支持从 GitHub 安装社区开源技能。

### 三级技能体系

| 级别 | 目录 | source 字段 | 优先级 |
|------|------|-------------|--------|
| 分身专属 | `avatars/<id>/skills/` | `local` | 最高 |
| 自有公共 | `shared/skills/` | `shared` | 中 |
| 社区开源 | `shared/skills/community/` | `community` | 最低 |

同名技能按 `local > shared > community` 优先级加载，分身可覆写任何公共或社区技能。

### 安装社区技能

1. 编辑 `shared/skills/sources.yaml`，添加技能源：
   ```yaml
   sources:
     - name: awesome-prompt
       repo: https://github.com/example/awesome-prompt-skills.git
       ref: v1.0.0
   ```
2. 运行同步脚本：`./scripts/soul-sync.sh`
3. 在分身的 `skills/skill-index.yaml` 中引用安装的技能

### 发布自己的技能

1. 创建 GitHub 仓库，按 `templates/skill-template.md` 格式编写技能文件
2. 在仓库根目录放置 `skill-manifest.yaml`（参考 `templates/skill-manifest-template.yaml`）
3. 发布 tag 作为稳定版本

## 任务拆分规则

> **首要判定**：收到任何用户请求时，**先用本节"触发判定"判断是否进入拆分流程**，再决定后续行为。
> 不满足触发条件时直接执行/直接回答，**不要输出"任务拆分计划"打断对话节奏**。

### 触发判定（必须先做）

✅ **触发拆分（输出计划等待确认）** — 同时满足以下两点：
1. 涉及**代码改动**（创建/修改/删除 `.ts` / `.tsx` / `.js` / `.py` 等源代码文件）
2. 任务规模属于"中任务"以上（涉及 ≥ 2 个源代码文件，或单文件改动 ≥ 10 行）

⛔ **不触发（直接执行/直接回答）** — 命中任一即可：
- 调研、分析、对比、解释、咨询类问题（"为什么 / 怎么样 / 是什么 / 对比 / 评估 / 推荐"）
- 文档查询、知识检索、产品咨询（分身桌面端对话场景默认走这条）
- 单文件改动 < 10 行
- 用户已经明确指定"改哪个文件 + 改什么"（如"把 foo.ts 第 12 行的 var 改成 const"）
- 改动仅涉及非代码文件（`.md` / `.json` / 配置文件 / CLAUDE.md / 知识文件等）
- 任何不涉及代码修改的对话

### 触发示例对照

| 用户输入 | 是否触发 | 原因 |
|---|---|---|
| "帮我实现一个用户认证模块" | ✅ 触发 | 涉及多文件代码新增 |
| "重构 chatStore.ts 把 sendMessage 拆开" | ✅ 触发 | 单文件大改动（架构层面） |
| "给 conversation-router 加 3 个 regex" | ⛔ 不触发 | 单文件小改动，直接执行 |
| "为什么我的 AI 分身不像 Claude Code" | ⛔ 不触发 | 调研/解释类问题，直接回答 |
| "给我对比一下 280Ah 和 315Ah 的能量密度" | ⛔ 不触发 | 知识检索，直接回答 |
| "解释一下 useEffect 的依赖数组" | ⛔ 不触发 | 解释类问题 |
| "改一下 README 的标题" | ⛔ 不触发 | 非代码文件 |

### 触发后：阶段一 — 输出拆分计划

1. 分析任务涉及哪些文件
2. 识别依赖关系（哪个先做，哪个后做）
3. 拆分为独立的子任务列表
4. 评估每个子任务的复杂度和风险
5. **输出拆分计划，等待用户确认**

输出格式：
```
## 任务拆分计划

原始需求：xxx

### 子任务列表
- [ ] 子任务 1：做什么 → 涉及文件：src/xxx.ts（预计 5 行修改）
- [ ] 子任务 2：做什么 → 涉及文件：src/yyy.ts（预计 10 行修改）
- [ ] 子任务 3：做什么 → 涉及文件：src/zzz.ts（预计 3 行修改）

### 执行顺序
1 → 2 → 3（原因：xxx 依赖 yyy）

### 风险提示
- 子任务 2 改动较大，建议先备份

确认后我将从子任务 1 开始执行。
```

### 触发后：阶段二 — 逐步执行

每次只执行一个子任务：

1. 执行前声明：**"当前执行：子任务 X / 共 Y 个"**
2. 只加载该子任务涉及的文件，不读取无关文件
3. 完成后输出进度和下一步
4. **等待用户回复"继续"或子任务编号后才执行下一个**

### 任务规模判断标准

| 规模 | 判断标准 | 处理方式 |
|---|---|---|
| 小任务 | 改动 < 10 行，只涉及 1 个源代码文件 | 直接执行，无需拆分 |
| 中任务 | 涉及 2-3 个源代码文件 | 拆分为 2-3 个子任务 |
| 大任务 | 涉及 4 个以上源代码文件 | 必须拆分，并分批确认 |
| 超大任务 | 跨模块/涉及架构改动 | 先出方案文档，再逐步执行 |

### 上下文控制规则

- 每个子任务只加载当前需要的文件
- 禁止在子任务中顺带"顺手改"其他文件
- 每完成 3 个子任务，建议新开对话继续（避免上下文过长）

### 熔断规则

- 单个子任务执行超过 2 轮仍失败 → 停止，标记为"待人工处理"
- 不得为了完成子任务而扩大修改范围
- 遇到阻塞时输出原因和建议，等待用户决策

### 自检（每次响应前默念）

> "用户这次的请求，落在'触发判定'的 ✅ 还是 ⛔ ？
> 如果是 ⛔，我直接回答/执行；如果是 ✅，我先输出拆分计划。
> 不要因为'看起来是大事'就强行拆分一个调研类问题。"

## TypeScript 质量检查规则

### 核心原则
- 先诊断，再修复
- 每次只处理一个文件
- 超过 3 轮未解决，停止修改，只分析原因
- 不得同时修改多个文件

### 阶段一：诊断模式（不修改任何代码）

当用户说"检查质量"、"分析问题"、"质量扫描"时：

1. 阅读用户提供的扫描报告（tsc / eslint 输出）
2. 将所有问题归类为：类型错误 / 代码规范 / 性能问题 / 架构问题
3. 按修复难度排序输出清单（简单 / 中等 / 复杂）
4. 每条问题注明：文件路径 + 行号 + 问题描述
5. **输出清单后停止，等待用户确认，不主动修改代码**

### 阶段二：修复模式（逐文件处理）

当用户说"修复第X条"或"处理 xxx.ts"时：

1. **只修改指定的单个文件**
2. 修改前先说明：打算改什么，为什么这样改
3. 修改后输出 diff 摘要：改了哪几行，改了什么
4. 询问用户是否继续下一条

禁止行为：
- 不得在修复一个文件时顺带修改其他文件
- 不得在用户未确认时连续修复多个问题
- 不得删除用户未提及的代码

### 死循环熔断机制

如果同一个问题已经修改超过 3 次仍未解决：

1. **立即停止修改代码**
2. 输出根本原因分析（已尝试次数、可能根因、建议方向）
3. 不再尝试新的修复，等待用户决策

### 推荐使用前的准备命令

```bash
# 生成类型错误报告
npx tsc --noEmit 2>&1 | head -100

# 生成 ESLint 报告
npx eslint . --ext .ts,.tsx --format compact 2>&1 | head -100
```
