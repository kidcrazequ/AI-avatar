# 决策回溯 · decision-trace

> **级别**：[■■■] 高阶
> **版本**：v1.0
> **最后更新**：2026-05-14

---

## 适用问题特征

用户问题命中以下任一形态时，**必须先加载本技能再回答**：

| 形态 | 例子 |
|---|---|
| 为什么 X 没做 Y | 「为什么 262 ODM2.0 没有使用远景 L05 Pack？」 |
| 为什么选了 Z 而不是 W | 「为什么 ODM2.0 沿用 1P52S 而不是 1P104S？」 |
| 这个决策怎么定的 | 「丝印颜色怎么定的红色？」 |
| 当时为什么取消了 X | 「当时为什么取消了第二道防凝露？」 |
| 谁拍板的 / 谁讨论过 | 「这个尺寸调整谁拍板的？」 |

**反例**（不属本技能，走 query-product-quality 或常规检索）：
- 「262 机型 1 月设备侧效率」（数据查询，不是回溯）
- 「L05Pack 容量多少」（参数题，不是回溯）

## 核心流程（5 步，禁止跳步）

### 步骤 1：候选文件清单（search_knowledge × 2）

并行调两次 `search_knowledge`，覆盖**两个独立检索维度**：

```js
// 维度 A：以产品/方案名 + 阶段为关键词
search_knowledge({ query: "<产品> <项目阶段> 变更点 评审" })
// 维度 B：以被询问的对象 + "评估 / 是否使用 / 讨论" 为关键词
search_knowledge({ query: "<被询问对象> 评估 是否使用 讨论" })
```

> **失败处置**：任一次 `search_knowledge` 返回过长（触发 spool 落盘提示），**必须**用 `read_tool_result` 读 spool 完整文件，不要凭头/尾摘要作答。`read_lines` 会因路径不在工作区而失败，必须用 `read_tool_result`。

预期产物：一份候选文件清单（典型 3-5 份），按相关性排序。

### 步骤 2：识别表格类决策记录（query_excel）

候选清单中如果有「**输入确认清单**」「**评审记录**」「**变更点表**」「**讨论纪要**」**任一**，**必须**用 `query_excel` 精确取行——这类表格的关键证据（拍板人答复、技术要求、风险注释）通常在备注列，被 search_knowledge 的摘要算法切掉。

```js
// 先看 schema 知道有哪些列、哪些 sheet
query_excel({ file: "<文件名>", schema: true })
// 再按"涉及对象"过滤取行（不要 columns 限制，让备注列出来）
query_excel({
  file: "<文件名>",
  sheet: "<sheet>",
  filter: { "<可能列名>": { "$contains": "<被询问对象>" } }
})
```

### 步骤 3：旁证（search_knowledge × N）

把步骤 2 拿到的决策点（如"防凝露""配置不统一"）作为新关键词，回查 search_knowledge 找**佐证文件**（用户手册、采购侧资料、其他项目同类讨论）。这一步证明"决策不是孤立的"。

### 步骤 4：综合证据，结构化回答

回答按下述**强制结构**输出（违反结构判定为偷懒）：

```markdown
**<阶段> 对 <被询问对象> 做过专项评估，最终 <结论>，核心考量如下：**

**1. <考量点 1 名称>**
<具体证据：人名 / 料号 / 原文片段 / 数值>。<决策含义>。

**2. <考量点 2 名称>**
...

**3. <考量点 3 名称>**
...

（一般 3-5 点；少于 3 点说明检索不够深）

综合以上因素，<阶段> <最终采用方案>，避免 <被规避的风险>。
```

### 步骤 5：来源声明

必须在末尾列出**具体文件名**（不是 `_excel/*.json` 中间产物路径）：

```
来源：
· `<原始 Excel 文件名>.xlsx`（<拍板人>逐行答复）
· `<项目变更点文件>.docx`（同期结构课题清单）
· `<用户手册或采购资料>.docx`（旁证）
```

**禁止用 `_excel/*.json` 路径作为来源**——那是内部中间产物，用户看不懂。

## 反模式（禁止）

下列回答模式会被判定为**人格失败**：

❌ **泛词替代原文**：「L05 Pack 侧重高能量密度，262 ODM2.0 针对短时调峰」——没有具体料号 / 决策人 / 项目阶段。
❌ **只列 1-2 个考量**：决策回溯类问题至少 3 个考量，少于 3 点说明步骤 1-3 没走通。
❌ **只用 search_knowledge 不用 query_excel**：备注列证据丢失，回答会浮于表面。
❌ **来源指 `_excel/*.json`**：暴露内部缓存路径，应映射回原 xlsx 文件名。
❌ **跳过 read_tool_result 直接读 spool**：用 `read_lines` 会路径穿越失败。

## 标准回答样例（参照案例二，**输出格式必须对齐**）

问：为什么 262 ODM2.0 没有使用远景 L05 Pack?

```
ODM2.0 阶段对 L05Pack 做过专项评估，最终未直接采用、沿用 1P52S 模组方案，核心考量如下：

**1. 防凝露设计风险**
L05Pack 没有单独的防凝露设计。堵杰与尹博确认：有除湿器的情况下可不再单独设计，但意味着除湿器一旦异常没有第二道防线。

**2. 配置不统一**
L05Pack 在大储侧有不同总成料号、配置不一（已知料号 3071900：熔丝中贝 ESH3155 + 十堰 315Ah + 远景 ADI 版 BMU(1818) + 含 Pack 消防）。换型号意味着工商储侧需重做适配。

**3. 二级物料未确认**
工商业储能柜单次需求量下，L05Pack 二级物料的价格、付款条件、交付周期响应仍待采购确认，无法形成稳定供应保障。

**4. 结构联动改动量大**
更换 PACK 会连带消防、液冷管路、BMS、线束、界面划分都要调整；同期 ODM2.0 还要做电气舱与电池舱彻底隔离、IP55 防护可行性等结构课题，并联多项重大变更会拉长项目周期、放大风险。

综合以上因素，ODM2.0 阶段沿用了原 1P52S 模组方案，避免在密集结构变更窗口内引入电芯/PACK 平台的不可控变量。

来源：
· `ODM2.0_262户外柜改制-产品信息输入确认清单 - Leon.xlsx`（堵杰逐行答复）
· `ODM2.0项目变更点.docx`（同期结构课题清单）
· `ENS-L262用户手册 ODM2.0 V2-2026.2.26 修正B版.docx`（沿用 1P52S 的间接证据）
```

注意上述样例的语料密度：**每一点都含具体料号 / 人名 / 数值**，没有任何"产品定位"式空话。

## 与其他技能的关系

- 调用前置：无（用户问题命中触发即可加载）
- 调用后置：如果回答需要画图或对比，可串 `load_skill('chart-from-knowledge')`
- 互斥：与 `query-product-quality` 互斥（那个是数据查询，本技能是决策回溯）
