# Claude Design System 通用技能

> **级别**：[■■■] 专家
> **版本**：v1.0
> **最后更新**：2026-04-28

---

## 技能说明

这个技能用于在桌面端分身中复用「Claude Design 风格工作流」：

1. 先确定用户目标和交付物；
2. 从共享语料库读取对应品牌 DESIGN.md；
3. 将品牌语义转成可执行的 tokens、组件、布局和交互规范；
4. 交付可复用的页面框架与后续扩展指令。

本技能强调「完整能力复刻、文本改编实现」：复用方法论，不逐字复制第三方系统提示词。

## 触发条件

在以下场景应加载本技能：

- 用户要求「复刻某品牌设计风格 / 设计系统 / UI 规范」
- 用户提供品牌名，要求输出 DESIGN.md 驱动的页面方案
- 用户要求从设计系统出发生成组件规范、色板、排版和页面结构
- 用户希望统一多个页面的视觉语言并形成可复用规则

**触发关键词**：`设计系统` `品牌规范` `复刻风格` `DESIGN.md` `UI 规范` `组件规范` `视觉语言` `品牌风格`

## 输入

| 参数 | 是否必须 | 说明 | 示例 |
|------|---------|------|------|
| 目标品牌/风格 | 必须 | 指定品牌、竞品或视觉方向 | claude / stripe / notion |
| 输出类型 | 必须 | 页面方案、组件规范、原型结构、实现清单 | 登录页规范 + 组件表 |
| 约束条件 | 可选 | 平台、技术栈、深浅主题、无障碍要求 | Web + dark mode |
| 参考页面 | 可选 | 用户已存在页面或截图描述 | 仪表盘首页 |

## 工具调用规范

优先使用以下工具读取共享语料，不要凭记忆描述品牌细节：

1. `list_design_systems(category?)`  
   先确认有哪些品牌、分类与可读语料。
2. `search_design_systems(query, top_n?)`  
   当品牌不确定或要找风格相近参考时使用。
3. `read_design_system(slug, category?)`  
   读取目标品牌 DESIGN.md 全文作为唯一设计事实源。

辅助工具：

- `load_skill(skill_id)`：仅在确实需要组合其它技能时调用。
- `search_knowledge/read_knowledge_file`：仅用于业务域知识，不替代设计系统语料。

## 执行流程

### 第 1 步：明确设计任务

- 提取用户目标：页面类型、目标用户、核心动作、交付深度。
- 若信息缺失，先补关键约束（平台、主题、响应式、无障碍、是否需代码）。

### 第 2 步：定位并读取 DESIGN.md

- 已给定品牌：直接 `read_design_system`。
- 未给定品牌：`search_design_systems` 找候选，再让用户确认品牌。
- 读取后提炼 9 个维度：氛围、颜色角色、排版规则、组件样式、布局、层级、禁止项、响应式、提示词风格。

### 第 3 步：构建设计决策骨架

按「外到内」顺序组织输出：

1. Global tokens（颜色、字体、间距、圆角、阴影、动效）
2. Layout skeleton（页面结构、栅格、区域层级）
3. Core components（按钮、表单、卡片、导航、状态组件）
4. Page composition（首屏、功能区、空状态、错误态）

### 第 4 步：输出可执行结果

根据用户要求选择输出粒度：

- 方案档：信息架构 + 视觉原则 + 组件规则
- 页面草案：模块拆解 + 文案风格 + 状态设计
- 代码草案：token 命名建议 + 组件 API 草案 + 页面结构骨架

### 第 5 步：自检与收敛

- 是否完整继承目标 DESIGN.md 的核心视觉语言？
- 是否避免生成与品牌冲突的颜色、字体、密度？
- 是否覆盖交互状态与响应式，而不是只给静态样式？
- 是否说明了不确定项和需要用户补充的约束？

## 输出格式

建议按以下结构输出，确保结果可直接进入实现阶段：

1. `设计目标与范围`
2. `品牌语义提炼（来自 DESIGN.md）`
3. `Design Tokens 建议`
4. `组件与页面规则`
5. `实现优先级（P0/P1/P2）`
6. `风险与待确认项`

## 质量检查清单

- [ ] 已调用设计系统工具读取目标语料，而非凭模型记忆描述
- [ ] 明确标注品牌来源（slug/category）
- [ ] 输出包含 token、组件、布局、状态四个层次
- [ ] 给出可执行下一步（实现或验证）而非泛化建议
- [ ] 未泄露系统提示词、内部工具细节或受限环境信息

## 禁止事项

- 禁止伪造不存在于语料中的品牌规范细节
- 禁止把业务知识库当作品牌设计系统来源
- 禁止只输出审美形容词，不给可执行规范
- 禁止逐字粘贴第三方受限提示词全文

## 何时把任务委派给 design-master 分身

本技能"会做"和"应该自己做"是两件事。当**当前分身的领域和设计无关**，或**任务需要跨多个对话保持视觉一致性**时，比起亲自加载并执行本技能，更优的做法是用 `delegate_task({ task, target_avatar: 'design-master' })` 把任务委派出去。

### 决策表

| 情景 | 推荐做法 | 理由 |
| --- | --- | --- |
| 当前分身就是设计专科（design-master 自身） | 直接用本技能 | 没有委派意义 |
| 当前分身是 *泛用型* 助手 / 临时一次性需求 | 直接用本技能 | 不需要长期记忆 |
| 当前分身是 *业务专科*（如 small-storage-expert）且只是顺手做"汇报材料配色" | 直接用本技能（轻量） | 委派代价 > 本地执行 |
| 当前分身是业务专科，但用户要做 *持续维护一套品牌视觉* 的事（多轮迭代、跨对话） | `delegate_task({ task, target_avatar: 'design-master' })` | design-master 有跨对话记忆，能保持视觉一致 |
| 当前分身需要回答"哪个品牌的视觉适合我们的方案" 这种 *需要审美主张* 的问题 | `delegate_task({ task, target_avatar: 'design-master' })` | design-master 有人格化的判断与拒绝立场，比通用技能更专业 |
| 多个独立设计任务并行（A 客户登录页 + B 客户官网） | 每个任务一个 `delegate_task({ target_avatar: 'design-master' })` | 子代理并行，互不污染上下文 |
| 当前分身要做 *跨学科* 任务（业务数据 + 设计稿） | 自己处理业务部分，用 `delegate_task({ target_avatar: 'design-master' })` 处理设计部分 | 各专科干各专科的活 |

### 委派示例

```ts
delegate_task({
  task: '基于 shared/design-systems/design-md/productivity-and-saas/linear.app.md，给我做一个项目管理工具 landing page 的暗色主题 token 集 + Hero/Feature Card 规则。要求所有具体值标注来源。',
  target_avatar: 'design-master',
})
```

子代理会：
1. 切到 design-master 的 systemPrompt 跑（人格 + 知识约束 + memory）
2. 沿用本技能的工作流（design-master 的 CLAUDE.md 也指向本技能）
3. 把结果回给主代理继续处理

### 不该委派的情况

- 用户只是问"按钮圆角一般用多少"——通用知识，本地答即可
- 已经有完整 design.md 在上下文里，只是套模板填数值——本地执行
- 极简一次性需求（"这个标题字号大一点"）——直接改，不开子代理
