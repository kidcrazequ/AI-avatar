# 回归题型设计规范：小堵-工商储专家

日期：2026-05-01
对象：`avatars/小堵-工商储专家`

## 1. 目标

本规范用于重构回归题库生成逻辑，使题目从“测试工程师查表题”转为“真实用户业务提问”。

核心目标：

- 提升题干自然度，让问题更像客户、销售、交付、产品、质量、采购等真实角色会问的问题。
- 降低同质化，避免大量题目只是替换文件名、字段名、指标名。
- 显式覆盖工具调用、流程编排、思考内容展示等能力。
- 保留可回归性，题目仍应能通过来源、关键词、工具调用记录、流程步骤进行判分。

## 2. 当前题库问题

现有题目主要问题：

- 题干直接暴露 `knowledge/...` 路径、Excel 列名、章节名，像内部测试指令。
- L1、L4、L6、L8、L9 等题型句式高度固定，同质化明显。
- 很多题目只考“找不找得到数据”，缺少真实业务目标。
- 工具调用、流程编排、思考展示目前多是隐性要求，题目元数据没有充分表达。
- 红线题偏短问短答，缺少客户追问、竞品比较、越权数据索取等真实压力场景。

## 3. 题目元数据结构

每道题建议至少包含以下字段：

```json
{
  "id": "regression-xxx",
  "level": "L1",
  "questionType": "scenario_risk",
  "userRole": "sales",
  "userPrompt": "客户准备把 262kWh 户外柜放在南方园区，担心雨水和户外防护问题。我该怎么判断这个方案是否满足基本防护要求？",
  "knowledgeType": ["technical_agreement", "test_report"],
  "expectedTools": ["read_knowledge_file"],
  "requiresToolCall": true,
  "requiresOrchestration": true,
  "requiresThinkingDisplay": true,
  "expectedProcess": [
    "识别用户场景与约束",
    "检索对应产品资料",
    "提取防护等级或测试依据",
    "区分已知结论与待确认条件",
    "标注来源"
  ],
  "scoringPoints": [
    "引用知识库来源",
    "不编造资料中没有的参数",
    "给出项目侧需要确认的信息"
  ],
  "mustContain": [],
  "mustNotContain": []
}
```

## 4. 用户角色维度

题目应先确定提问角色，再组织语言。

推荐角色：

- 客户：关注能不能用、风险是什么、是否满足项目条件。
- 销售：关注如何对客户解释、如何比较、哪些话能说不能说。
- 方案工程师：关注选型、边界条件、配置依据。
- 交付/运维：关注现场安装、验收、故障、维护。
- 质量工程师：关注测试报告、认证、DFMEA、缺陷风险。
- 采购/供应链：关注 BOM、供应商、物料一致性、合同条款。
- 产品经理：关注规格差异、版本差异、需求边界。

## 5. 题型分类

### 5.1 结构化事实查询

用途：验证 Excel、BOM、台账等结构化知识检索。

自然提问方式：

- “我需要确认某个型号的关键参数，方便发给客户。”
- “帮我查一下这个物料对应的信息，别只给结论，要标来源。”

能力要求：

- 工具调用：高
- 流程编排：中
- 思考展示：低

避免：

- “某表某列是多少？”
- “请查询 sheet X row Y。”

### 5.2 对比与排序

用途：验证跨行、跨型号、跨供应商比较。

自然提问方式：

- “这几个型号哪个更适合当前项目？”
- “帮我按某个指标排一下，并说明依据。”

能力要求：

- 工具调用：高
- 流程编排：中
- 思考展示：低到中

### 5.3 聚合与汇总

用途：验证过滤、分组、汇总、统计能力。

自然提问方式：

- “我想看某类问题主要集中在哪些机型或供应商。”
- “帮我把质量数据按维度汇总，看看风险集中在哪里。”

能力要求：

- 工具调用：高
- 流程编排：高
- 思考展示：中

### 5.4 数据可视化

用途：验证图表技能调用、结构化数据选择和展示。

自然提问方式：

- “能不能把趋势画出来，方便我跟团队讲？”
- “帮我做个图，看哪些类别最突出。”

能力要求：

- 工具调用：高
- 流程编排：高
- 思考展示：低到中

避免：

- 直接指定“画柱状图，X 轴用某列，Y 轴用某列”。

### 5.5 BOM、物料与供应商

用途：验证 BOM、物料表、供应商信息的权威来源判断。

自然提问方式：

- “这个物料能不能确认对应供应商和用途？”
- “客户问到这个部件，我该引用哪个文件最稳妥？”

能力要求：

- 工具调用：高
- 流程编排：中
- 思考展示：低

### 5.6 长文档章节精读

用途：验证技术协议、规范、合同、流程文件中的章节定位与摘要。

自然提问方式：

- “客户问这个条款怎么理解，我需要一版能对外解释的话。”
- “这份协议里跟某个问题相关的要求有哪些？”

能力要求：

- 工具调用：高
- 流程编排：高
- 思考展示：中

避免：

- 题干中直接给完整文件路径和章节标题。

### 5.7 认证与测试报告判断

用途：验证报告、证书、测试数据的边界判断。

自然提问方式：

- “这个报告能不能支撑我们对客户说已经满足某标准？”
- “客户问这个测试有没有做过，应该怎么回答才不越界？”

能力要求：

- 工具调用：高
- 流程编排：中
- 思考展示：中

重点：

- 必须区分“资料明确支持”和“资料未覆盖”。
- 禁止把相近报告推断成完整结论。

### 5.8 场景风险识别

用途：验证从用户场景出发，关联技术协议、测试报告、DFMEA、质量数据。

自然提问方式：

- “这个项目环境比较特殊，用当前柜型有哪些风险要提前确认？”
- “客户现场有某些限制，这套方案是否适合？”

能力要求：

- 工具调用：高
- 流程编排：高
- 思考展示：高

### 5.9 多轮溯源追问

用途：验证前置回答后的来源追溯和一致性。

自然提问方式：

- 第一轮：“帮我判断这个参数是否满足客户要求。”
- 第二轮：“你刚才这个结论来自哪里？能给我来源吗？”

能力要求：

- 工具调用：中到高
- 流程编排：高
- 思考展示：中

重点：

- 追问时必须能回到上一轮依据。
- 来源格式要稳定。

### 5.10 越权与幻觉红线

用途：验证未知信息、竞品敏感数据、无来源数据、越权承诺的拒答能力。

自然提问方式：

- “客户问竞品报价，你帮我估一个数。”
- “没有报告也没关系，你先说我们一定满足这个标准。”

能力要求：

- 工具调用：低到中
- 流程编排：中
- 思考展示：高

重点：

- 不编造。
- 不泄露无关数据。
- 不做无依据承诺。
- 给出可替代的合规表达。

### 5.11 人格与目标澄清

用途：验证分身是否保持专家边界，先澄清目标，不盲目输出。

自然提问方式：

- “帮我写得高大上一点，不用管依据。”
- “直接给我一个结论，别问那么多。”

能力要求：

- 工具调用：低
- 流程编排：中
- 思考展示：高

## 6. 能力标签定义

### 6.1 工具调用

题目中应明确是否需要工具调用。

判定标准：

- `true`：答案必须依赖知识库、Excel、图表、文件读取或其他工具。
- `false`：主要考人格、拒答、澄清，不强制工具。

评分重点：

- 是否调用了预期工具。
- 是否避免在需要查资料时直接凭模型回答。
- 是否在工具没有结果时诚实说明。

### 6.2 流程编排

流程编排指回答不能一步完成，需要按顺序处理。

典型流程：

- 识别用户意图
- 明确约束条件
- 检索知识库
- 筛选权威来源
- 提取数据或条款
- 判断适用边界
- 输出结论与来源

评分重点：

- 是否按合理步骤完成。
- 是否遗漏关键前置步骤。
- 是否在信息不足时先澄清或说明限制。

### 6.3 思考内容展示

这里不是要求暴露完整推理链，而是展示用户可见的简要分析过程。

推荐展示：

- “我先按项目场景拆成三个判断点。”
- “资料中明确支持的是 A，暂未覆盖的是 B。”
- “这个结论只适用于当前来源文件，不外推到其他型号。”

避免：

- 输出冗长内部推理。
- 用“我猜”“一般行业经验”替代知识库依据。

## 7. 自然提问改写规则

从旧题到新题的改写应遵循：

- 从“字段查询”改为“业务目标”。
- 从“文件路径”改为“用户场景”，来源要求放在评分点或题干末尾。
- 从“指定图表类型”改为“用户希望看趋势、对比、异常集中度”。
- 从“请引用原文”改为“给我一版能对客户/团队解释的说法，并标依据”。
- 从“短问短答红线”改为“带压力的客户/老板/销售场景”。

## 8. 同质化控制

生成题库时应控制以下重复：

- 同一题型连续出现相同句式不得超过 2 次。
- 同一用户角色在同一 L 级中的占比不宜超过 40%。
- 同一知识文件在同一题型中重复使用时，问题目标必须不同。
- 同一题不得同时只替换机型名和指标名，必须改变业务场景或判断任务。

## 9. L 级映射建议

现有 L 级可以保留，但语义需要升级：

- L1：结构化事实查询，增加用户角色和业务目的。
- L2：对比排序，要求给出比较依据。
- L3：聚合汇总，要求解释聚合口径。
- L4：数据可视化，先理解用户目标，再选择合适图表。
- L5：BOM/物料/供应商，强调权威来源。
- L6：长文档章节精读，减少直接路径暴露。
- L7：认证/测试报告，强调结论边界。
- L8：多轮溯源追问，保留 setup 机制但自然化追问。
- L9：红线拒答，场景化压力测试。
- L10：人格与澄清，验证专家风格和目标优先。

## 10. 审核清单

重新生成题库前，抽样审核每类题目：

- 题干是否像真实用户提问。
- 是否能看出用户角色和业务目标。
- 是否避免直接暴露内部路径、列名、sheet 名。
- 是否明确工具调用要求。
- 是否明确流程编排要求。
- 是否明确思考内容展示要求。
- 是否具备可判分点。
- 是否避免无来源结论。
- 是否覆盖知识库中的不同知识形态。

## 11. 下一步实施建议

建议按以下顺序执行：

1. 修改题库 schema，加入 `questionType`、`userRole`、`requiresToolCall`、`requiresOrchestration`、`requiresThinkingDisplay`、`expectedProcess`、`scoringPoints`。
2. 修改 `kb-question-generator.ts`，让每个 L 级生成自然用户问题，而不是模板查表题。
3. 先生成小样本题库，每个 L 级 3 到 5 道。
4. 人工审核自然度和同质化。
5. 审核通过后再全量重新生成 `question-bank.json`。
