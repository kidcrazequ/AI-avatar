- ✅ **决策**：原 #13「Chunk 可视化面板（S~M, 3-5 天）」按 2026-05-09 决议**降级**为「CLI 轻量版（XS, 0.5 天）」。落地为 `desktop-app/scripts/knowledge-inspect.ts`（370 行）+ `desktop-app/package.json` 加 `knowledge:inspect` script（与 `backfill:life` 同 lane）+ `desktop-app/docs/knowledge-inspect-cli.md`（147 行用户文档）。**零新依赖**（未 `npm install` 任何包）；**零改 core**（`packages/core/src/knowledge-retriever.ts` git diff 为空）。
- ✅ **决策**：CLI 故意走 `KnowledgeRetriever.getFullChunks()` 而非 `searchChunks()`。原因：`searchChunks` 内置 `MIN_CHUNK_LENGTH=80` + `isMetadataChunk` 双重过滤（`knowledge-retriever.ts:323-325`），是 RAG 召回路径；本工具是**诊断**工具，要看到 `buildChunks` 真实切出的全集（含被过滤的短 chunk），才能反映 chunking 改动效果。同样**不**用 `warmUpAsync()`：异步路径（193-216 行）未跳过 Excel `rag_only` 大表 .md，与默认同步 `buildChunks` 行为不一致，会输出失真。

**指挥官-W15-#11-2026-05-09（#11 Scheduled Tasks 用户自定义定时任务）**：
- ✅ **决策**：方案为**扩展现有 CronScheduler**而非另起新调度模块。原因：Soul 已有 `cron-scheduler.ts`（230 行）+ `schedule-cron` IPC，承担 `memory-consolidate` / `knowledge-check` / `scheduled-test` / `life-advance-all` 4 类内置任务的 `setInterval` / `scheduleDailyCallback` 调度；硬剥离会引发 4 处 callsite 同步迁移与 `scheduledTester` / `life-advance-all` 重写。最终在同一个 `CronScheduler` 内**新增独立的 `cronJobs` Map** 与 `scheduleCron` / `cancelCron` / `getNextRuns` / `cancelAllCron` / `hasCronTask` 5 个方法（基于 `croner@^10.0.1`），与既有 `timers` / `dailyCallbacks` Map **完全隔离**；`cancelAll` 内部联动清理。
- ✅ **决策**：触发链路用 `trustTier='proxy'` 经 `proxyJobId` 注入（沿用 #1/#7 已落地的代码路径 `chatStore.ts:2284-2286`），**不**引入第三种 trustTier。原因：避免在 `tool-permission-policy` 增加新分支，调度任务的"无人值守"语义与"代理外部触发"在权限模型上同源；`proxyJobId` 命名为 `schedule:<id>:<firedAtUtc>`，便于审计。
- ✅ **决策**：`schedules` 与 `schedule_runs` 双表（v9→v10 迁移），用 `UNIQUE(schedule_id, fired_at_utc)` 提供**幂等约束**（cron 自然触发 + UI「立即触发」可能撞车）。`recordRunStart` 捕捉 SQLITE_CONSTRAINT_UNIQUE 静默返回 `conflict: true` 跳过。`FOREIGN KEY ON DELETE CASCADE` 保证删除 schedule 时历史日志同步清理。
- ✅ **决策**：错过触发**不补跑**（开机时 `restoreSchedulesFromDb` 仅注册 enabled 任务，不扫描历史 `next_run_at`）。原因：避免一次性涌入历史消息，造成 token 突发。文档明确说明此 policy；用户需要补跑请人工点「立即触发」。
- ✅ **决策**：DAO 独立成 `db-schedules.ts`（`ScheduleStore` 接 `Database.Database` 注入），**不**塞进 `DatabaseManager`。原因：① `DatabaseManager` 已逼近 1000 行；② 单测可不依赖 electron stub，直接 in-memory db 注入；与 `ConversationJsonlAppender`（#2）同款解耦设计。代价：`DatabaseManager` 暴露 `getRawDb()` getter 给 `ScheduleStore`/`main.ts` 拼装时使用，仅限 `desktop-app/electron/*` 内部调用。
- ✅ **决策**：UI 提供「简易（每天 / 每周 / 每月 / 每 N 小时）+ 高级（cron 表达式）」双模式，`cronstrue@^3.14.0` 中文释义实时显示，下 3 次触发由主进程 `schedule:get-next-runs` 计算（避免渲染端二次实现 cron 解析）。
- ✅ **回归**：tsc --noEmit 0 错误；`db-schedules.test.ts`（12 用例）+ `cron-scheduler.test.ts`（9 用例）+ `database-schedules-migration.test.ts`（3 用例）共 **24 用例全部 pass**（Node ABI 兼容时）；ABI 不兼容时优雅 skip 15 项 SQLite 用例 + 跑通 9 项纯 cron 用例。Lint 改动文件全绿。
- ✅ **影响下游**：未来若有任务想「让分身定期主动报告」（如 #12 ASR 后的"每日语音日报"），可直接复用 `schedule:create` IPC + 现有 `schedule-trigger-handler.ts`，不需要再造调度。
