# 设计决策表检索（先查表，再出方案）

> **级别**：[■■] 进阶
> **版本**：v1.0
> **最后更新**：2026-07-05

---

## 技能说明

出任何设计方案之前，先按行业/场景检索 `shared/design-systems/design-decision-table.csv`
命中的规则行，方案逐条对照 must_have 与 anti_pattern，并引用行号或
category+industry_or_context 作为来源。决策表是从 shared/design-systems 73 套品牌语料中
提炼的结构化规则集（借鉴 ui-ux-pro-max 的 CSV 决策表机制）；技能本身只是薄检索层，
规则的正本在 CSV 与其 source 列指向的语料原文里。

## 触发条件

在以下场景应读取并遵循本技能：

- 用户要求输出任何设计方案 / 视觉方向 / 设计系统 / 页面设计
- 用户询问某行业或产品类型"该长什么样"（色彩、字体、布局、组件、深浅色取舍）
- 用户要求评审 / 改版一个现有设计（对照 anti_pattern 做体检）

**触发关键词**：设计方案、视觉方案、风格、配色、字体、布局、组件规范、深色模式、
落地页、官网、UI、UX、design system、palette、typography、圆角、阴影、品牌感

## 输入

| 参数 | 是否必须 | 说明 | 示例 |
|------|---------|------|------|
| 行业 / 场景 | 必须 | 用于检索 industry_or_context 列 | 金融科技、企业级 B2B、内容媒体深色应用 |
| 风格取向 | 可选 | 极简 / 浓郁 / 编辑式 / 深色优先等 | 极简开发者工具 |
| 品牌参照 | 可选 | 命中后可回读 source 列语料原文取细节 | "想要 Stripe 那种感觉" |

用户未说明行业/场景时，先追问一句再查表，不凭默认审美直接出方案。

## 决策表结构

`shared/design-systems/design-decision-table.csv`，7 列：

| 列 | 含义 |
|---|---|
| category | 色彩 / 字体 / 布局 / 组件 / 深度层次 / 响应式 / 动效 / 信息架构 |
| industry_or_context | 行业或 if_X 触发条件（如 "if 深色模式优先产品"） |
| recommendation | 推荐做法，具体到值（hex / px / 权重） |
| must_have | 硬性要求（可为空） |
| anti_pattern | 反模式，点名到具体值（可为空） |
| severity | red（红线，违反即否决）/ yellow（明显偏离）/ info（风格建议） |
| source | 语料原文路径（相对 shared/design-systems/），规则可回溯的正本 |

## 执行流程

1. **提取检索词**：从用户需求中提取行业、产品类型、深浅色取向、风格关键词。
2. **检索决策表**：用已有检索能力（search_knowledge 的 BM25，或直接读 CSV 后按
   industry_or_context / category 关键词过滤行）。不引入任何外部脚本。
3. **列出命中行**：给出行号（CSV 数据行号）+ category + industry_or_context 摘要。
   跨行业通用行（如浅色界面正文色、焦点可达性）与行业专属行都算命中。
4. **逐条对照后出方案**：方案中每个关键决策（色彩 / 字体 / 布局 / 组件 / 深度层次）
   对照命中行的 must_have 与 anti_pattern；severity=red 的行绝不违反；违反 yellow 行
   必须显式说明理由。需要比 CSV 一行更细的规格（完整色板、字阶表）时，回读该行
   source 列指向的语料原文。
5. **标注来源**：每条被采纳的规则标 `[决策表 L<行号>: <category>/<industry_or_context>]`；
   回读语料的细节标语料文件路径。
6. **无命中则明说**：决策表未覆盖该行业/场景时，明确说明"决策表无命中，以下为
   最接近的相邻场景行 + 通用行"，列出缺口，不假装有依据。

## 输出格式

方案正文之后附一张决策对照表：

| 设计决策 | 命中行 | must_have 核对 | anti_pattern 核对 |
|---------|--------|---------------|------------------|
| 强调色 #0f62fe 单色策略 | L2 色彩/企业级 B2B | ✅ 交互色唯一 | ✅ 未引入第二强调色 |

## 反模式（禁令 + 归因 + 后果）

| 禁令 | 归因（为什么禁） | 违反后果 |
|------|----------------|---------|
| 禁止不查决策表直接出方案 | AI 惯性：统计均值审美（紫粉渐变 hero、万物 12px 圆角、默认 Inter + 蓝紫强调色）正是决策表要封杀的输出 | 方案退化为均值审美，B6 机制失效，方案作废重做 |
| 禁止凭通用设计常识编造规则再标 CSV 行号或语料来源 | 溯源红线同型事故（2026-05-22 来源错位）：把非表内内容标成表内来源 | 触发数据溯源红线，整段回答判废 |
| 禁止命中 severity=red 的行仍继续输出违反方案 | 模型在用户压力下会 rationalize（"用户就想要多彩一点"）；red 行来自品牌级红线（单强调色、语义色独立、品牌红每屏唯一） | 一票否决，方案不可交付 |
| 禁止把命中的 anti_pattern 软化成"也可以考虑" | anti_pattern 点名到具体值就是为了不可协商；软化 = 变相违反 | 对照表流于形式，等同没查 |
| 禁止用一句"参考了设计决策表"盖整个方案 | 一行来源盖整段是溯源粒度禁令的翻版 | 每个关键决策必须能定位到行号级 |

## 交付前自检

- ☐ 已检索 CSV 并在输出中列出命中行（行号或 category+industry_or_context）
- ☐ 方案的每个关键决策（色彩/字体/布局/组件/深度层次）至少对照一行，或显式声明"决策表未覆盖"
- ☐ 无 severity=red 违反；每个 yellow 违反都写了显式理由
- ☐ 引用的行号在 CSV 中真实存在，行的 source 列可回溯到 design-md 语料文件
- ☐ 输出中没有"业界通常 / 一般来说 / 大家都这么做"式的无来源论断

## 不适用范围（转介给谁）

- 不做数据图表设计（柱状/折线/饼图）：转 `shared/skills/draw-chart.md`
- 不做单一品牌整套 DESIGN.md 复刻 / token 体系生成：转 `shared/skills/claude-design-system.md`
- 不做交互原型 / 动画视频等制品产出：转 `shared/skills/claude-interactive-prototype.md` 等制品技能（本技能只负责方案上游的规则校验，可在制品技能之前串联使用）
- 决策表未覆盖的行业（如医疗、政务）：给出最接近的相邻场景行并明确列出缺口，不硬套

## 引用知识

执行本技能时，优先参考以下文件：

- `shared/design-systems/design-decision-table.csv` — 决策表正本（检索目标）
- `shared/design-systems/design-md/<分类>/<品牌>.md` — 各行 source 列指向的语料原文（细节回读）
- `shared/design-systems/INDEX.md` — 73 套语料的目录索引
