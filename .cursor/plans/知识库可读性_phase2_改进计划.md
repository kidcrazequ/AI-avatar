## 二、子任务列表

### 子任务 1：`prepareTable` 表头检测增强（解决 P1）

| 项 | 内容 |
|---|---|
| **类型** | 源码修改 |
| **涉及文件** | `desktop-app/electron/document-parser.ts` — `prepareTable()` 方法 |
| **预计代码量** | 40-60 行 |

**根因分析（来自 Phase 2 探查）**：

30 个 col 文件分 3 类：

| 类型 | 示例 | 原因 | 占比 |
|---|---|---|---|
| A. 封面行占据前 5 行 | `通用柜体检验指导书.md` | 前 2 行是标题+文件编号，真表头在第 3-4 行但含空 cell | ~40% |
| B. 表单型无表头 | `远景能源标准箱单.md`、`包装确认单.md` | 整个 Excel 是填写型表单，根本没有"列名行" | ~40% |
| C. 宽表部分空列 | `BOM_L262.md` | 表头行存在但个别列为空 | ~20% |

**改动方案**：

```
1. 扫描深度从 5 → 10（覆盖封面后的真表头）
2. 新增"封面行跳过"启发式：
   - 若某行首 cell 有值但其余 ≥80% 为空，且该行在前 3 行内
   → 标记为封面/标题行，不参与打分
3. 新增"两行表头合并"：
   - 若得分最高的行仍有 ≥30% 空 cell，检查其下一行
   - 若下一行能补全空 cell → 合并两行作为表头
   （解决"适用部门 | col2 | col3"+"序号 | 检验项目 | ..."的两行表头问题）
4. 对类型 B（表单型），保持 col1..colN fallback 不变（无法改善，这类文件本身就没有表头）
```

**预期效果**：30 个文件中 ~18 个（A+C 类）获得更好表头，B 类 ~12 个维持现状。

**回归风险**：改 `prepareTable` 影响所有 Excel 转出，必须用回归脚本验证 `_excel/*.json` 不变。

### 子任务 2：keywords 提取规则扩展（解决 P2 - keywords 部分）

| 项 | 内容 |
|---|---|
| **类型** | 源码修改 |
| **涉及文件** | `packages/core/src/utils/knowledge-frontmatter.ts` — `extractKeywords()` |
| **预计代码量** | 30-40 行 |

**根因分析**：

当前 `extractKeywords` 仅从 2 个来源提取：
1. 产品编码正则 `[A-Z]{2,5}-[A-Z0-9][-A-Z0-9_]{2,}` — 只覆盖英文编码
2. Markdown 标题 `^#{1,3}\s+(.{2,25})$` — 很多文件无 markdown 标题（纯表格/纯流式文本）

**改动方案**：

```
新增 3 个 keywords 来源（按优先级）：

3. 中文产品/文档关键词：
   - 正则匹配常见行业术语：储能|电芯|模组|BMS|PCS|EMS|液冷|柜体|检验|DFMEA|PPAP 等
   - 从 bodyText 前 3000 字中提取

4. 表格首列高频值：
   - 对 Excel 源文件，扫描第一列非空值中频次 ≥2 的短文本（≤10 字）
   - 例：调试问题 → "CAN通讯", "BMS-PCS", "BMS-EMS" 等

5. frontmatter 已有字段衍生：
   - 从 `raw_file` 文件名分词提取（去除数字/版本号/日期后的中文片段）
   - 例："远景能源262kWh工商业储能柜技术规范书" → ["远景能源", "工商业储能柜", "技术规范书"]
```

**预期效果**：keywords 覆盖率从 33% 提升到 ~75%。

### 子任务 3：summary 提取规则扩展（解决 P2 - summary 部分）

| 项 | 内容 |
|---|---|
| **类型** | 源码修改 |
| **涉及文件** | `packages/core/src/utils/knowledge-frontmatter.ts` — `extractSummary()` |
| **预计代码量** | 15-25 行 |

**根因分析**：

当前 `extractSummary` 跳过标题、表格、分隔线，寻找"首段有意义文本"。但很多文件（Excel 源/表单型 PDF）正文直接就是表格，前 30 行内无段落文本。

**改动方案**：

```
1. 增加 fallback 策略（当前 30 行内无段落文本时）：
   a. 尝试取第一个 markdown 标题作为 summary 前缀
   b. 尝试取 "> 导入自 Excel/PDF" 行的信息
   c. 尝试取表格首行前 3 列的拼接内容
   d. 最终 fallback：从文件名清洗生成一句描述

2. 扫描范围从 30 行 → 50 行（部分文件正文在更后面开始）
```

**预期效果**：summary 覆盖率从 33% 提升到 ~80%。

### 子任务 4：`<br>` 标签优化处理（解决 P3）

| 项 | 内容 |
|---|---|
| **类型** | 源码修改 + 一次性脚本 |
| **涉及文件** | `desktop-app/electron/document-parser.ts` — `rowsToMarkdownTable()` + 新脚本 |
| **预计代码量** | 40-60 行 |

**根因分析**：

Excel 单元格内的换行符 `\n` 在 `rowsToMarkdownTable` 中被转为 `<br>`，导致：
- 表格列挤压为超长单行
- 渲染后可读性差，尤其多条检验要点串在一起

**改动方案**：

```
方案 A（推荐）：智能拆分
- 当单元格含 ≥2 个 <br> 时，改为编号列表格式
- 例：`1.门锁开合<br>2.开启角度<br>3.两扇门之间`
  → `1. 门锁开合\n2. 开启角度\n3. 两扇门之间`
- 在表格外用缩进 blockquote 展示

方案 B（保守）：<br> → 分号
- 将 `<br>` 替换为 `；` 分隔
- 简单粗暴但不破坏表格结构

建议先实现方案 B（源码改动），再用一次性脚本处理存量 41 个文件。
```

**一次性脚本**：`desktop-app/scripts/cleanup-br-tags.ts`
- 扫描所有 .md 中表格行的 `<br>` 标签
- 按方案 B 替换为 `；` 或 ` / `
- 支持 `--dry-run` / `--apply`
