**W3-#3-2026-05-09（#3 完成）**：
- ✅ **决策**：仅在 `tools.length > topN` 时启用；`topN` 默认 15、范围 5–64（设置项 `iss_skill_rerank_top_n`）；关闭开关写入 `iss_skill_rerank_enabled=false`
  - **原因**：与 §4.3 AnythingLLM 默认对齐；阈值与 topN 合并为「计数大于 topN 才重排」避免重复配置
- ✅ **决策**：embedding 调用走渲染进程 **`fetchWithTimeout`**（DashScope-compatible `/embeddings`），与 **`electron/llm-factory#createEmbeddingFn`** 请求体对齐；向量缓存 **`localStorage` key `soul.iss.skill-embeddings.v1`**（hash→向量 JSON，最多约 640 条淘汰）
  - **原因**：`@soul/core/browser` 无 fs；与知识索引同源 embedding 模型 `text-embedding-v3` / 512 维复用 OCR 槽位密钥
  - **影响下游**：验收「token 降 30%+」仍需补 API `usage` 解析或离线对比脚本（当前仅 `bodyChars`/toolCount 启发式）；无 OCR Key 时自动跳过 ISS 并重试全流程工具表
- ✅ **决策**：钉住 **`list_mcp_tools` / `call_mcp_tool` / `todo_write` / `load_skill`** 始终进入 LLM tools，不参与淘汰
  - **原因**：MCP 网关与编排工具被误判相似度会造成任务不可用
  - **影响下游**：#7 Permission Mode 若改工具名集合需同步 `ISS_DEFAULT_PINNED_TOOL_NAMES`（或届时改为配置项）

**W2-#2-2026-05-09（#2 完成）**：
- ✅ **决策**：JSONL 路径约定为 `<userData>/conversations/<conversationId>.jsonl`，单会话一个文件
  - **原因**：与 §4.2 设计决策一致；离线脚本可直接按文件名定位，conversationId 冗余写入每条记录便于跨文件解析
  - **影响下游**：#8 结构化白盒 Memory 可参考此命名约定（`<userData>/memory/<avatarId>.jsonl`？由 #8 自定）
- ✅ **决策**：写入时序 = SQLite `saveTx()` 提交成功 → 事务外 fire-and-forget `void appender.append(...)`
  - **原因**：appender 内部已 try/catch + warn 兜底，Promise 不会 reject；不 await 保持 saveMessage 同步签名（不破坏现有调用方）；append 异步 `fs.promises` 不阻塞 IPC 主线程
  - **影响下游**：所有后续在 saveMessage 旁挂副作用的任务都应遵循此模式（事务外 + fire-and-forget + 内部兜底）
- ✅ **决策**：失败处理风格采用 `logger.warn` 而非 `console.warn`，并通过最小接口 `JsonlAppenderLogger { warn(msg, err?) }` 解耦
  - **原因**：Logger 类只有 `error()` / `logEvent()`，没有 `warn()`；声明结构化最小接口便于注入 fake 单测，main.ts 一行 lambda 适配 `logEvent('warn', ...)`
  - **影响下游**：electron-main.mdc 规则「只有 Logger 自身兜底允许 console.warn」继续保持；后续若有同类需求可参考此 lambda 适配模式
- ✅ **决策**：DatabaseManager 构造函数 `(dbPath?, jsonlAppender?)` 第二参数可选，向后兼容
  - **原因**：`getDb()` lazy fallback 路径不传 appender，备份缺失在异常恢复路径上可接受；新增的 main.ts 正常初始化路径才注入
  - **影响下游**：未来若想严格全覆盖，可在 main.ts 抽 `getOrInitJsonlAppender()` 辅助函数
- ⚠️ **环境提醒**：仍未触碰 better-sqlite3 ABI 问题（appender 单测不依赖 SQLite，纯 fs，跑测试无需 rebuild）；database.ts 改动仅靠 tsc --noEmit 静态验证（0 错误）+ 手工核对接入位置正确，运行时验证留待全量回归。

**W1-#5.5-2026-05-09（#5.5 完成）**：
- ✅ **决策**：`createBaseSchema()` 末尾追加 `mcp_servers` 表定义，字段与 v5→v6 migration 逐字对齐（采用方案 A）
  - **原因**：v5→v6 migration（行 448-469）保留不动，保护老用户升级路径；`CREATE TABLE IF NOT EXISTS` 保证幂等
  - **影响下游**：解除 #2 SQLite+JSONL 双写对 `database.ts` 的串行等待
- ✅ **决策**：项目实际测试栈是 `node:test` + `tsx --test`，**不是 vitest**
  - **原因**：原计划文件提到 vitest，但仓库实际用 `node:test`（参考 `database-attachments.test.ts`）；本次新测试已对齐现状
  - **影响下游**：后续所有 #2 / #3 / #4 等任务的测试编写都应采用 `node:test` 风格，避免再次踩坑
- ⚠️ **环境提醒**：`better-sqlite3` 默认按 Electron ABI 编译，本地直接用系统 Node 跑测试会被 ABI 探测优雅 skip。如需真实跑通需 `npm rebuild better-sqlite3 --build-from-source`，跑完务必还原（避免 dev 启动失败）。本次已还原。建议未来 CI 用 Electron 自带 Node 跑测试，避免该步骤。
